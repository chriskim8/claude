## Tone Guidance

- Write as if speaking to a smart, non-technical colleague — not to a developer
- Use **bold** to call out UI labels, button names, and page paths exactly as they appear in the product
- Avoid: ticket IDs, internal code terms, raw error messages unless quoting for clarity
- If a bug is involved: acknowledge the issue plainly, give the workaround, don't over-explain the technical cause
- If a workaround exists: lead with it, then explain the permanent fix is coming
